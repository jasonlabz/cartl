# Cartl 功能页逐页视觉与交互迁移执行计划

## 目标与工作方式

- 来源：用户要求以 `dagine-dashboard/web` 为唯一基线，将 `cartl` 的功能页按页面逐个对照、逐个修改；登录页最后处理。
- 成功标准：每个已验收页面在约定视口下，与 Vue 基线的壳层、信息结构、间距、控件状态、表格、弹窗/抽屉及现有业务操作一致；验证通过后才开始下一页。
- 工作方式：不先做全站统一翻新。每一波只修改一个页面及该页不可避免的共享样式；对照结果经浏览器验证并记录后锁定，再进入下一波。

## 前置事实与 Gates

- 变更范围：frontend（`cartl` React）；`dagine-dashboard/web`、后端、数据库与 API 契约均不改。
- 技术栈：React 19、TypeScript strict、Vite、React Router、Ant Design；Vue 基线使用 Vue 3 与 Naive UI。
- 视觉基线：Vue 主题为 `#646cff` 主色、`220px` 侧栏、`56px` Header、`44px` 标签栏、`#f7fafc` 内容背景；功能页默认有 `16px` 内容留白。
- 登录联调：开发代理已修复 `/proxy-default`，可使用用户提供的测试账号登录；本计划不记录账号或密码。
- 工作区：`cartl` 存在既有未提交迁移改动；仅触碰本计划列出的文件，绝不还原、清理或覆盖无关改动。

## 对照与验收协议

每个页面均按以下顺序执行，未通过不进入下一页：

1. 固定对照条件：登录同一测试用户，清空页面临时筛选；分别在 `1440px`、`1280px`、`768px`、`390px` 视口打开同一路由。
2. 先记录 Vue 基线：页面壳层尺寸、标题/工具栏、筛选字段、按钮、表格列与分页、空态/加载态、抽屉或弹窗、关键操作路径。
3. 仅修改该 React 页面及必要的共享组件/主题样式；共享改动必须同时回归已验收页面。
4. 在相同条件下逐项对照 React 与 Vue；视觉通过后执行关键交互（搜索、重置、分页、创建/编辑/删除或该页等价操作）。
5. 运行静态、单元和构建检查，记录差异、验证结果和已知限制；将该页标为“已锁定”。

“1:1”按可观测行为验收：不将浏览器字体渲染、Ant Design 与 Naive UI 的不可消除亚像素差异视为功能偏差；颜色、尺寸、层级、布局、可操作状态和业务结果必须一致。

## 执行策略

- 方式：串行。页面会共同依赖 `BasicLayout`、主题 token、表格和页面容器；并行修改会互相覆盖视觉决定。
- 首个试点：`/standards/org`（Vue 名称：机构编码规范）。它是完整但边界清晰的 CRUD 列表页，覆盖筛选、表格、分页、弹窗和删除确认，适合先校准公共样式且不涉及复杂画布。
- 共享改动原则：首页只提取经 Vue 对照确认的最小公共规则；不得借机重写其他页面或将未验证的视觉假设推广到全站。
- 波次准入：前一波的视觉检查、关键交互和 `pnpm` 验证均通过，且差异已记录或修复，才能开始下一波。

## 首个页面实施细化：机构编码规范

### 目标

将 React 的 `/standards/org` 与 Vue 的 `src/views/standards_org/index.vue` 在功能和视觉上对齐，并把已验证的“功能页列表”基础规则限定为后续页面可复用的基线。

### 范围内

- Vue 对照：`dagine-dashboard/web/src/views/standards_org/index.vue`、Vue 布局与主题文件。
- React 实现：`cartl/src/pages/standards/org/index.tsx`、其直接引用的样式/通用表格/页面容器；只有页面无法正确对齐时才修改基础布局或 token。
- 测试：为新增或改变的可复用行为补聚焦 Vitest 测试；现有测试不得回退。

### 范围外

- 不改登录页、后端、Vue 源码、API service 契约、其他功能页的业务 DOM 或文案。
- 不提前迁移元数据、工作流画布、算子库、调度中心。

### 验收点

- 壳层：220px 侧栏、56px Header、44px 标签栏及 16px 内容留白与 Vue 对齐。
- 页面：筛选区与操作区的字段、顺序、间距、按钮语义与 Vue 对齐；不保留营销式英文 eyebrow 或大标题。
- 表格：列、状态标签、行操作、加载/空态、分页位置与密度对齐。
- 交互：查询、重置、分页、新增、编辑、启停（如 Vue 提供）、删除确认与数据刷新结果一致。
- 响应式：在 768px、390px 不遮挡、不出现不可达操作。

## 执行顺序说明

1. 先完成任务 0 和 1，交付可直接预览的机构编码规范页面，而不是只交付设计说明。
2. 用户可在每个“已锁定”页面后提出像素级修正；修正只影响当前页和已验证的共享规则。
3. 后续按照任务 2 至 8 的模块顺序推进；若用户指定另一页面为下一优先级，则在不破坏已锁定基线的前提下调整下一波顺序。

## 风险与回滚

- 风险：React/Ant Design 与 Vue/Naive UI 的默认控件样式不同。缓解方式是每页先对照后抽取最小 token，避免一次性全局覆盖。
- 风险：共享布局调整可能影响已完成页面。缓解方式是每次共享改动均回归此前已锁定页面。
- 风险：复杂工作流的后端 payload 与画布状态不同步。缓解方式是将其独立为任务 4，先做行为回放再做视觉收口。
- 回滚：每个波次只涉及有限的 React 文件，可按波次的变更集回退；不涉及数据、后端或 Vue 源工程。
