# Cartl Vue -> React 全站重构技术方案

## 1. 背景与目标
`dagine-dashboard/web` 是基于 Vue 3、Naive UI、Pinia、Vue Router、Vue Flow 的数据治理控制台。`cartl` 是其独立 React 重构项目。目标是在不改 Vue 源工程和后端契约的前提下，完成全站 React 迁移，并以 `http://localhost:9527/home` 为视觉基线做现代化优化。

## 2. 范围与非目标
### 范围
- 应用启动、登录、错误页、布局、菜单、动态权限路由、主题、国际化、全局反馈。
- 标准管理、数据工坊、质量中心、数据洞察全部可见页面和详情/版本页。
- 工作流画布和 Transform Workbench 的行为等价迁移。
- API service、类型、错误处理、鉴权 token、分页和状态反馈迁移。

### 非目标
- 不修改 `/Users/lucas/workspace/code/golang/dagine-dashboard/web`。
- 不改变后端路由、请求字段、响应字段、成功码、过期码和权限语义。
- 不在 React 迁移期间删除原 Vue 工程；最终切换前保留回滚入口。

## 3. 项目规则与技术栈判断
- 目标项目已初始化为 React 19 + TypeScript strict + Vite + React Router 7。
- 按 fullstack-flow React 规范组织 `src/layouts`、`src/routes`、`src/pages`、`src/components`、`src/services`、`src/types` 和 `mock`；菜单 key、路由 path、页面目录保持对齐。
- 组件库统一收敛到 React 侧组件体系，优先 Ant Design 5/6 及项目需要的 antd-style；已有 Lucide 图标可作为图标补充。视觉 token 和原 Vue 主题值集中管理。
- 请求层使用 axios `BaseRequest` 风格统一封装，业务 API 方法统一 `xxxApi` 命名；请求错误、鉴权过期和全局提示在拦截器处理。
- 状态优先使用 React Context + Hooks；页面查询参数集中到 `tableParams` / `listParams`，异步请求采用请求序号或 AbortSignal 防止旧请求覆盖新状态。
- 工作流画布使用 `@xyflow/react`，纯业务模型和转换逻辑与 React 组件分离，避免把 Graph 状态直接作为后端 payload。

## 4. 原 Vue 代码映射
| Vue 现有边界 | React 目标边界 | 迁移要求 |
|---|---|---|
| `src/main.ts` / `App.vue` | `src/main.tsx` / `src/App.tsx` | Provider、错误边界、路由入口等价 |
| `src/layouts/**` | `src/layouts/**` | BasicLayout、BlankLayout、主题和标签页拆分 |
| `src/router/**` | `src/routes/**` | 静态/动态权限、菜单 meta、详情路由对齐 |
| `src/store/modules/**` | `src/stores/**` 或 Context hooks | auth、app、route、tab、theme 状态等价 |
| `src/service/api/**` | `src/services/**` | 原 API path、参数、响应和错误语义保持 |
| `src/components/flow/**` | `src/pages/workflow/flow-designer/components/**` | 画布、节点、算子和工作台分层迁移 |
| `src/views/**` | `src/pages/**` | 菜单、路由、页面、service、mock 五层对齐 |

## 6. 状态与数据流
### 全局状态
- `AuthContext`：token、用户信息、角色和登出。
- `AppContext`：移动端、侧栏、滚动和布局设置。
- `RouteContext`：权限路由、菜单、面包屑、缓存路由。
- `ThemeContext`：明暗模式、主题色、布局和水印。
- `TabContext`：多标签页、缓存和关闭/刷新行为。

### 页面状态
- 列表统一使用 `{ page, page_size, keyword, filters... }` 单对象管理查询参数。
- 请求函数独立封装，useEffect 仅依赖查询参数触发；操作成功后通过函数式更新查询参数刷新。
- 复杂详情页采用 `loading / error / data` 三态，异步轮询或竞态请求使用 AbortSignal/序号。

### 工作流模型
- `FlowDocument`：节点、边、全局配置和 revision。
- `FlowCanvasState`：React Flow 节点/边以及选择、视口和临时 UI 状态。
- `WorkbenchState`：按 node/target 组织的配置切片。
- `flow-adapter`：在持久化模型和 React Flow 模型之间做双向转换和校验。

## 7. API 与鉴权兼容
- API 契约不变，因此不进入 `api-doc`；迁移前逐个对照原 `src/service/api/*.ts`。
- `GET/POST/PUT/PATCH/DELETE` 方法、`/api/v1` 前缀、分页字段 `list/total`、业务 code、token refresh 和 logout codes 全部保持兼容。
- 原 Vue request transform 要拆为 React request interceptor + response normalizer；不在页面内重复处理成功码或 token 过期。
- React dev server 继续代理 `/api`，生产部署使用同源 API 或配置化 base URL。

## 8. 视觉和交互方案
- 原首页用于对照基线：侧栏宽度、菜单层级、卡片密度、指标、图表、数据源健康度和质量概览保留其工作台语义。
- React 侧统一使用 token：颜色、字体、间距、边框、阴影、状态色、圆角和响应式断点，避免每个页面独立调色。
- 组件级样式放在页面/组件目录，禁止继续把所有业务样式写入单一 `src/styles.css`。
- 表格、表单、抽屉、弹窗、空态、错误态和 loading 统一从公共组件/主题层提供。
- 动效只用于路由进入、抽屉和状态反馈；不使用营销式 hero、过度渐变或装饰性背景。

## 9. 分阶段迁移与兼容
1. 基础设施：目录、请求层、共享类型、Provider、路由和视觉 token。
2. 低风险页面：登录、错误页、组织机构、值域、质量规则/方案/报告/工单。
3. 数据资产页面：数据源、资产仓库、元数据及详情/版本、数据目录/探查/对比。
4. 运行页面：算子库、调度、任务列表/详情、首页真实数据联调。
5. 高风险工作流：数据流列表、Flow Canvas、Transform Workbench、保存/校验/回填。
6. 切换与收口：端到端回归、视觉对照、性能、权限、错误恢复和发布切换。

## 10. 安全、权限和回滚
- React 不扩大权限；菜单可见性和路由守卫以原角色/按钮权限为准。
- 不在前端日志、mock 或文档中写入真实 token、连接配置或敏感数据。
- 原 Vue 入口保留；每个迁移波次可按路由回切，React 失败不影响原系统运行。

## 11. 验证方案
- 静态：`pnpm typecheck`、`pnpm lint`（补齐后）、`pnpm build`。
- 单元：纯转换器、payload builder、权限过滤、分页和错误映射。
- 交互：登录、菜单/标签、列表筛选分页、表单提交、抽屉/弹窗、画布节点和保存。
- 浏览器：原 Vue `9527` 与 React `cartl` 对照检查 1440、1280、768、390 视口。
- 性能：首屏 chunk、图表、画布节点规模、长列表和路由切换响应。

## 12. 未决问题
- 生产切换是按路由反向代理、独立子路径还是一次性替换，需发布阶段确认。
- Ant Design/antd-style 的最终主题映射需要以原 Vue 主题截图和产品评审为准。
- 真实 API 环境、登录账号和权限矩阵需要联调环境提供；未具备时先使用脱敏 mock。
