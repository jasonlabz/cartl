# Cartl Vue -> React 全站重构需求分析

## 输入来源
- 用户需求：遵循 fullstack-flow React 规范，在 `cartl` 中实现，并以原 Vue 首页 `http://localhost:9527/home` 为基础优化样式。
- 参考实现：`dagine-dashboard/web` 的 Vue 首页、导航、主题、路由与 API 契约。
- 目标项目：`/Users/lucas/workspace/code/golang/cartl`。

## 当前状态
- 需求属于前端 React 全站重构，规模判定为 L。
- 已确认 React 19、React Router 7、Vite、React Flow、Recharts、Lucide 技术栈。
- 原项目包含 27 个业务视图目录、150 个 Vue SFC、148 个 TypeScript 文件；`src/components/flow` 约 20,562 行。
- 当前 `cartl` 是 React 高保真原型，主要实现集中在 `src/App.tsx`，尚未达到全站业务等价。
- 用户要求以 `http://localhost:9527/home` 的原页面作为视觉基线，优化 React 版本而不是弱化原有后台信息密度。

## 规模判定理由
- 命中 L 级综合条件：跨模块、复杂状态和数据流、工作流画布与配置工作台、组件库/路由/状态体系替换、全站视觉回归和长期双系统兼容。
- “没有 API 变更”只意味着跳过 `api-doc`，不降低页面迁移、交互复现和回归验证的规模。
- 预计按 5 个迁移波次交付，每波都需要构建、功能回归和桌面/移动视觉验收；不能以单个首页或 LOC 估算为 S。

## 功能范围
- 重建应用壳层：登录、错误页、布局、动态权限路由、菜单、标签页、主题、国际化、全局反馈和更新检查。
- 迁移工作台：首页、组织机构、值域标准、元数据标准及详情/版本页面。
- 迁移数据工坊：数据源、资产仓库、算子库、数据流列表/详情、调度中心、任务列表/详情。
- 迁移质量中心：规则、方案、报告和问题工单。
- 迁移数据洞察：资产目录、数据探查、数据血缘和版本对比。
- 迁移工作流核心交互：React Flow 画布、节点类型、算子面板、属性配置、转换工作台、字段映射、目标视角和保存/校验流程。
- 以原 Vue 页面为视觉基线，在 React 侧统一设计令牌、间距、表格、表单、抽屉、弹窗、图表和响应式行为。
- API 层保持与原项目 `/api/v1` 及鉴权/错误语义兼容，不改变后端契约。

## 非目标
- 不修改 `dagine-dashboard/web`。
- 不修改后端 API 契约、数据库或鉴权模型。
- 不在迁移过程中删除原 Vue 工程；采用并行部署和按路由切换策略。
- 不把现有 React 原型数据当作最终生产数据；所有业务页最终必须接入真实 service。

## 验收标准
1. 所有原有可见路由在 React 中有对应入口，关键路径、参数和返回行为保持一致。
2. React 首页以原 Vue 首页为视觉基线，桌面端信息密度、导航层级和关键卡片达到同等或更好的可读性。
3. 所有列表页具备 loading、empty、error、permission、搜索、筛选、分页和操作反馈状态。
4. 关键表单具备校验、提交中、失败重试、成功后刷新和关闭重置行为。
5. 工作流画布覆盖节点拖拽、连线、删除、配置、校验、保存、重载回填和冲突提示。
6. 角色权限、token 过期、动态路由和后端错误码沿用旧系统语义。
7. 1440px、1280px、390px 和 768px 视口下无内容遮挡、溢出或不可操作控件。
8. 每个迁移波次通过 `pnpm typecheck`、`pnpm build`、聚焦交互测试和浏览器回归后才切换路由。

## Gates
| Gate | 结果 | 处理 |
|---|---|---|
| DB schema | n-a | 无数据库变更 |
| code generation | n-a | 无生成代码 |
| permission | n-a | 沿用现状，未改变权限模型 |
| API contract | no change | 保留现有 `/api/v1` 路径 |

## 假设与风险
- 假设“在原来的基础上”表示以原 Vue 页面作为行为和视觉基线，代码继续落到 `cartl` React 项目。
- 原 Vue 与 React 在迁移期间并行存在，原工程只读；切换以路由/部署入口为边界。
- `cartl` Git 工作区已有大量用户删除项，本需求只改新增 React 文件和本需求文档，不恢复或覆盖无关改动。
- 原项目已有 28 个 node:test 文件但无统一前端 test script；React 侧需先建立可执行的测试和浏览器回归入口。
- 最大风险是工作流配置模型与画布交互同步，需最后迁移并进行数据回放对照，不能只做视觉仿制。
