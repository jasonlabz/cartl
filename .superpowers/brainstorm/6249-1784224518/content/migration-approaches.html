Dagine 全量 React 迁移：实施路径

目标不变：所有原路由、业务交互、后端 API 契约与视觉效果都迁入 cartl。选择的是交付节奏，不是功能范围。

迁移范围（全部保留）
基础能力
登录、注册、找回密码、403/404/500、鉴权、布局、主题、搜索
标准管理
组织、值域、元数据及详情、版本
工作流
数据源、资产仓库、算子、数据流、画布、任务、调度
质量管理
规则、方案、报告、问题工单
数据视图
目录、探查、血缘、对比
后端对接
沿用现有 /api 代理、令牌、响应与错误语义
基础能力 → 标准管理 → 工作流 → 质量 → 数据视图

A · 分域增量迁移（推荐）

先建共享壳与 API 层，再以业务域为单位逐个迁入、逐个进行视觉和交互回归；保留全量完成标准。

所有页面并行重写 → 集中联调 → 一次性交付

B · 集中式重写

视觉样式可能更早统一，但共享模式和 API 假设会在后期集中暴露，返工风险最高。

React 壳 → Vue iframe/微前端 → 逐页替换

C · 临时嵌入旧系统

短期切换快，但路由、主题、认证和视觉一致性会双重维护；不符合“React 重构”的最终目标。

建议选择 A：它将视觉回归和真实 API 联调置于每个业务域完成时，避免把风险集中到最后。
